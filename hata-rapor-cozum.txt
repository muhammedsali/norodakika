NOR ODAKİKA - HATA RAPORU VE ÇÖZÜM NOTLARI

 Kapsam
 Bu rapor; uygulamanın LocalStorage tabanlı yapıdan Firebase (Auth + Firestore) altyapısına taşınması, kimlik doğrulama akışları, veri yazma/okuma süreçleri ve çevrimdışı/opsiyonel servislerde (API vb.) hata toleransı sağlamak için yapılan düzenlemelerin özetidir.

 Notlar
 - Bu doküman “makale” değil; proje boyunca karşılaşılan hata tiplerini, kök nedenlerini ve kalıcı çözüm yaklaşımını kayda geçirmek için hazırlanmış teknik rapordur.
 - Bazı maddeler doğrudan gözlemlenen stacktrace’lerden değil, kodda yapılan koruma/iyileştirme değişikliklerinin işaret ettiği tipik arıza senaryolarından türetilmiştir.


 1) Firebase’e geçiş sonrası temel konfigürasyon eksikleri

 Belirti
 - Uygulama çalıştırıldığında Auth/Firestore çağrılarında PlatformException, missing configuration ya da “default Firebase app has not been initialized” benzeri hatalar.
 - Firebase özellikleri cihazda çalışmıyor; giriş/kayıt çalışmıyor.

 Kök neden
 - FlutterFire yapılandırmasının (firebase_options.dart üretimi / ilgili platform dosyaları) tamamlanmaması.
 - Android/iOS tarafında Google Services / plist vb. platform dosyalarının eklenmemesi ya da yanlış ortam (flavor) kullanımı.

 Çözüm
 - FlutterFire yapılandırması tamamlandı (beklenen adım: flutterfire configure).
 - Uygulama içinde Auth state Stream bazlı takip edilerek (AuthGate yaklaşımı) kullanıcı durumuna göre yönlendirme kurgulandı.

 Doğrulama
 - Temiz kurulum + uygulama açılışında kullanıcı durumunun akması (oturum yoksa auth ekranı, oturum varsa ana akış).
 - Kayıt/giriş sonrasında state’in anlık güncellenmesi.

 Öğrenimler
 - Firebase geçişlerinde “kod tarafı hazır” olsa bile, platform konfigürasyonu eksikleri en sık başlangıç problemidir.


 2) Google Sign-In (google_sign_in 7.x) initialize edilmeden kullanım

 Belirti
 - Google ile giriş denendiğinde giriş penceresi açılmaması veya runtime hatası.
 - Bazı sürümlerde GoogleSignIn örneğinin yanlış oluşturulması nedeniyle “initialize gerekli” kaynaklı problemler.

 Kök neden
 - google_sign_in 7.x ile birlikte singleton kullanımına ve initialize() çağrısına ihtiyaç duyulması.
 - serverClientId (Web client ID) tanımlı değilse Google token akışı başarısız olur.

 Çözüm (kod yaklaşımı)
 - AuthService içinde GoogleSignIn.instance kullanıldı.
 - İlk kullanım öncesi _ensureGoogleInitialized() eklendi.
 - serverClientId boşsa açık ve yönlendirici hata mesajı ile süreç durduruldu.

 Doğrulama
 - Google Sign-In akışı çalışıyor.
 - serverClientId eksik olduğunda anlaşılır hata veriyor.

 Öğrenimler
 - Sürüm geçişlerinde (major update) paket davranışları değişebilir; servis katmanında “initialize guard” eklemek stabiliteyi artırır.


 3) Kayıt (register) akışında “başarılı işlem sonrası hata fırlatıldı” senaryosu

 Belirti
 - Kullanıcı kayıt ol butonuna basınca UI’da hata görülmesi; ancak Firebase Console’da kullanıcı oluşmuş olması.
 - Kullanıcı tekrar kayıt denediğinde “email-already-in-use” gibi hata alması.

 Kök neden
 - Auth ile kullanıcı oluşturma başarılı olsa bile, sonrasında yapılan ikincil işlemler (Firestore’a profil dokümanı yazma gibi) fail olunca tüm register akışının hata olarak değerlendirilmesi.
 - Network dalgalanması / Firestore permission / geçici servis hatası gibi nedenlerle ikincil adımlar hata üretebilir.

 Çözüm (kod yaklaşımı)
 - Auth oluşturma ile Firestore profil yazma ayrıştırıldı.
 - Firestore’a profil yazma kısmı ayrı try-catch’e alındı; hata kritik değil kabul edilip sadece loglandı.
 - Ek olarak genel catch bloğunda “auth.currentUser != null ise” kullanıcı zaten oluştu varsayılıp işlem başarılı kabul edildi.

 Doğrulama
 - Kayıt sonrası kullanıcı oturumu açılıyor ve AuthGate üzerinden akış devam ediyor.
 - Firestore yazma sorunlarında kullanıcı giriş yapmaya devam edebiliyor.

 Öğrenimler
 - “Asıl kritik adım” (Auth oluşturma) ile “ikincil adımlar” (profil dokümanı, analytics, API kaydı) ayrılmalı.
 - Kullanıcı deneyimi açısından: “kayıt oldu ama hata gösterdik” senaryosu en maliyetli senaryolardan biridir.


 4) Firestore yazma/okuma hatalarının uygulamayı kilitlemesi

 Belirti
 - Oyun sonucu kaydı / profil güncelleme sonrası ekranın hata vermesi veya akışın kesilmesi.
 - Kimi cihazlarda geçici ağ problemi olduğunda kayıtların tamamen kaybolması.

 Kök neden
 - Firestore işlemleri (saveAttempt, updateGameDifficulty, saveUserData) ağ bağımlı ve permission kurallarına bağlı.
 - Tek bir Firestore hatasının tüm akışı kırması istenmiyor (özellikle oyun/attempt kaydı).

 Çözüm (kod yaklaşımı)
 - Oyun denemesi kaydında önce LocalStorage’a yazma garanti altına alındı.
 - userId 'guest' değilse Firestore’a yazma denendi; hata olduğunda akış kesilmeden loglandı.
 - Zorluk güncelleme gibi ikincil Firestore yazmaları da ayrı try-catch ile sarmalandı.

 Doğrulama
 - Ağ kapalıyken oyun oynama ve sonuçların en azından yerelde saklanması.
 - Ağ geri geldiğinde kullanıcı tekrar giriş yaptığında verilerin kaybolmaması (yerel history’nin varlığı).

 Öğrenimler
 - Offline-first yaklaşım: “yerelde sakla, bulut senkron opsiyonel” modeli mobilde daha dayanıklı.


 5) LocalStorage JSON parse (decode) hataları

 Belirti
 - Uygulama açılışında veya geçmiş ekranında history/stats okunurken hata logları.
 - Kullanıcı verisi bozulduğunda uygulamanın çökmesi.

 Kök neden
 - SharedPreferences içinde saklanan JSON’un bozulması, format değişimi, eksik yazım veya eski sürümden kalan veri.

 Çözüm (kod yaklaşımı)
 - Okuma tarafında jsonDecode try-catch ile sarmalandı.
 - Decode başarısız olursa boş liste/varsayılan değer dönülerek uygulamanın ayakta kalması sağlandı.

 Doğrulama
 - Bozuk veri senaryosunda uygulama çökmüyor; history boş kabul edilip devam ediyor.

 Öğrenimler
 - Mobilde local cache formatları zamanla değişir; backward-compatible okuma katmanı şarttır.


 6) Opsiyonel API (norodakika.com) entegrasyonunda hataların kritikleşmesi

 Belirti
 - Attempt submit ya da history/stats isteklerinde HTTP status code != 200/201 durumunda Exception.
 - API geçici olarak kapalıysa kullanıcı akışının bozulması riski.

 Kök neden
 - API endpoint’leri opsiyonel olmasına rağmen doğrudan throw edilmesi.

 Çözüm (kod yaklaşımı)
 - API submitAttempt çağrısı ayrı try-catch ile sarıldı; hata kritik değil kabul edildi ve sadece loglandı.
 - DailyPlan için API başarısız olursa MemoryBank üzerinden fallback üretildi.

 Doğrulama
 - API kapalıyken oyun akışı çalışmaya devam ediyor.
 - DailyPlan API başarısızsa uygulama boş kalmıyor.

 Öğrenimler
 - Opsiyonel servisler kullanıcı akışını kesmemeli; fallback stratejisi şart.


 7) Riverpod Async state + UI geri bildirim (Snackbar / navigasyon) senkronizasyonu

 Belirti
 - Kayıt/giriş sonrası snackbar’ların üst üste binmesi veya yanlış ekranda kalması.
 - Async state değişiminde mounted kontrolü yapılmadığında setState/navigasyon hataları.

 Kök neden
 - Async işlemler sonrası widget dispose olmuş olabilir.
 - Aynı event döngüsünde birden fazla snackbar tetiklenmesi kullanıcı deneyimini bozar.

 Çözüm (kod yaklaşımı)
 - UI tarafında mounted kontrolü kullanıldı.
 - Snackbar gösterimi öncesi clearSnackBars() ile önceki mesajlar temizlendi.
 - Başarılı kayıt sonrası AuthGate’e yönlendirilerek tek bir “otoriter karar noktası” oluşturuldu.

 Doğrulama
 - Hızlı ardışık denemelerde UI stabil; snackbar mesajları daha tutarlı.

 Öğrenimler
 - Auth gibi global state’lerde yönlendirmeyi tek bir gate ekranında toplamak hata ihtimalini düşürür.
